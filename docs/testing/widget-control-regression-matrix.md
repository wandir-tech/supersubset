# Supersubset — Widget And Control Regression Matrix

> **Purpose**: Provide a living, per-surface regression plan for every bundled analytical widget and user-facing filter control. Use this file when answering "have we actually proven this works?" and when deciding what commands must run before a widget or control change is considered done.

## Scope

This matrix covers:

- bundled dashboard widgets registered from `packages/charts-echarts/src/index.ts`
- content and control surfaces authored through Puck that affect dashboard behavior
- filter controls that can change analytical meaning for end users

This matrix complements, not replaces:

- `docs/testing/verification-strategy.md`
- `docs/testing/qa-checklist.md`
- `.github/skills/testing-strategy/SKILL.md`

## Maintenance Rules

1. If a PR adds, removes, renames, or materially changes a widget or control, update this file in the same PR.
2. If a PR changes any exposed property, default value, option set, field binding, or runtime behavior, update the corresponding row in the same PR.
3. Each row must maintain evidence at the right layers:
   - `U`: package or runtime unit proof
   - `D`: designer mapping or round-trip proof
   - `B`: browser proof for user-visible behavior
4. Do not call a row browser-certified unless a Playwright or browser workflow changes the property live and verifies the visible result.
5. Human-found bugs must update the affected row with the missing proof and add a named regression test before the issue is considered closed.

## Runbook

### When a widget or control changes

Run the narrowest commands that cover the touched rows:

- `pnpm --filter @supersubset/charts-echarts test`
- `pnpm --filter @supersubset/designer test`
- `pnpm --filter @supersubset/runtime test`
- `pnpm exec playwright test e2e/interactions/dashboard-filter.spec.ts e2e/workflows/filter-cascade.spec.ts --project=chromium`
- `pnpm exec playwright test e2e/workflows/designer-chart-matrix.spec.ts --project=chromium`
- `pnpm exec playwright test e2e/workflows/designer-to-renderer.spec.ts e2e/workflows/designer-page-management.spec.ts --project=chromium`

Add a narrower widget-specific browser spec when the existing browser proof does not actually exercise the changed property live.

### Release gate

Before claiming broad widget/control confidence on a release candidate or hardening branch, run:

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm test:e2e`

## Current Read

- The current base has strong `U` and `D` coverage for nearly all bundled widgets.
- The current base has meaningful browser proof for filters, including filter-bar title/layout/subset behavior, bundled charts, tables, KPI cards, markdown, alerts, and broad workflow health.
- The current base now includes live viewer proof for every bundled chart type exercised on the chart-gallery page, plus the overview-page line/bar, table, and KPI surfaces.
- The current base now includes live viewer proof that table `pageSize`, `showRowNumbers`, and `striped` changes survive designer-to-viewer serialization, and that KPI `suffix`, `fontSize`, and `comparisonField` changes visibly affect the rendered card.
- The current base now includes live browser proof that an authored page-scoped select filter changes overview-page widgets without leaking into detail-page widgets inside the same workbook.
- The current base now includes live viewer proof that horizontal bar charts render a vertical zoom rail when shared chart `zoomable` is enabled, closing the main shared-chart zoom gap for horizontal category axes.
- The current base now includes live browser proof that chart-gallery markdown content survives a designer-to-viewer round trip instead of regressing to `Unknown: markdown`.
- The current base does **not** yet have exhaustive live-property browser proof for every widget attribute, but every bundled chart type now has at least one viewer-backed property round-trip proof.
- The current base now has contract-closed proof for authored static select-filter option sourcing. Field-backed dynamic option resolution is still not exercised end to end in the runtime.
- The current base now includes authoring-surface proof that the `Dashboard Filters` drawer can open while a filter-bar widget is selected and that adding a filter scrolls/focuses the new editor instead of silently appending off-screen.
- The current base still lacks an explicit viewer unavailable state for field-backed authored filters when no host support exists; treat that path as a documented non-ready gap, not as covered end-to-end behavior.

### Dashboard Filter Release Rules

- Do not claim “dashboard filters are fully covered” unless both the authoring surface and the viewer/host analytical path are covered. Definition-only proof is insufficient.
- Do not count a browser pass as valid if the drawer only worked because the tester knew to ignore the filter-bar widget inspector. The shell must make the authoring surface legible enough for first use.
- Treat field-backed filters without runtime host support as a named gap until the viewer shows an explicit unavailable/loading/error treatment instead of an apparently functional empty control.
- When a human finds a dashboard-filter UX miss, update this matrix with the missing scenario row and add at least one lower-level regression plus one browser proof if the user can perceive the failure.

## Completion Rule

Do not answer "yes, this whole widget or control is proven" unless:

1. the relevant row is up to date,
2. the cited automated anchors still exist and pass,
3. the changed property has live browser proof if a user can perceive it,
4. the release gate commands are green on the target branch when claiming broad readiness.
